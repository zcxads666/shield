### 1.4 穿透根因分析

**Payload**: `1%u0027%20OR%20%271%27=%271`

**绕过原理**:
1. `%u0027` 是非标准URL编码（ASP/IIS风格），Go的 `url.QueryUnescape` 遇到 `%u0027` 会返回错误
2. `normalizeInput` 中的递归URL解码循环因错误而提前终止，导致 `%20`（空格）未被解码
3. 解码后的字符串变为 `1'%20OR%20'1'='1`，而非预期的 `1' OR '1'='1`
4. 现有正则模式无法匹配含 `%20` 的字符串，导致绕过

**影响范围**:
- 任何包含 `%u0027` + 其他百分号编码（如 `%20`, `%3b`, `%28` 等）的payload均可绕过
- 包括但不限于：OR/AND布尔盲注、UNION注入、时间盲注(SLEEP/BENCHMARK/pg_sleep)、堆叠查询

**验证示例**:
- `1%u0027%20OR%20%271%27=%271` → 404（穿透）
- `1%u0027OR(1)=(1)` → 403（拦截，因无其他百分号编码）
- `1%27%20OR%20%271%27=%271` → 403（拦截，标准编码无此问题）

## 五、改进建议

### 5.1 高优先级：修复 %u0027 解码绕过

**问题**: `normalizeInput` 中 `%uXXXX` 解码应在 `url.QueryUnescape` 之前执行，或采用更健壮的解码策略。

**建议修复方案**:
1. **方案A**: 在 `normalizeInput` 中先执行 `%uXXXX` 解码，再执行 URL 解码
2. **方案B**: 使用自定义的宽松URL解码器，跳过无效编码而非整体失败
3. **方案C**: 在 `collectParamsWithBody` 中增加对 `%u0027` 的预处理

**参考代码修改** (`internal/defender/sqlinject.go`):
```go
func normalizeInput(input string) string {
    s := input
    
    // 先解码 %uXXXX，避免其干扰后续 URL 解码
    s = normReUXXXX.ReplaceAllStringFunc(s, func(m string) string {
        r, _ := strconv.ParseUint(m[2:], 16, 32)
        return string(rune(r))
    })
    
    // 再执行递归 URL 解码
    for i := 0; i < 3; i++ {
        d, err := url.QueryUnescape(s)
        if err != nil || d == s {
            break
        }
        s = d
    }
    // ... 其余解码逻辑
}
```

### 5.2 中优先级：增强 UNION 无空格检测

当前 `\bUNION\b.*\bSELECT\b` 模式依赖单词边界，对 `UNIONSELECT` 无空格形式检测不足。

### 5.3 低优先级：CC阈值调优

同URL 120次访问中首次拦截索引为100，阈值（100次/60秒）合理但可结合实际业务场景微调。
